The Problem

A newly created diagnostics team with no established standard. Features had DFMEAs accounting for failures — but no structured framework defining how to detect, flag, and diagnose those failures from a diagnostics engineering standpoint.

The Approach

Executed a 5-phase program: system audit, gap analysis across legacy documentation, requirements creation, test case definition, and validation planning. Treated existing DFMEAs as the input — diagnostics requirements as the structured output.

The Standard

The 3-tier taxonomy (below) is now the working methodology for the team. What took months to develop for PRB's obstacle detection function — 68 requirements closing 12 gaps — becomes the template for every feature that follows.

Original Diagnostic Reasoning Taxonomy

Three requirement tiers, each building on the previous — from physics of failure to engineer-ready diagnosis. Authored from first principles.

Tier I
Detection Control
Defines the technical conditions under which a failure can occur — the physics and system-state boundaries that make a fault mode active. Sourced directly from feature DFMEA and ECU specifications.
Tier II
Fault Detectability
Defines how the vehicle perceives and flags the failure — what the driver or system experiences, and which DTC must be triggered. Translates engineering failure modes into observable vehicle behavior.
Tier III
Diagnostics Requirement
The engineer- and dealer-facing output: the full diagnostic path combining Tier I + II — specifying which DID to read, which signal to capture, and what the DTC confirms. Enables precise, repeatable troubleshooting.

"68 requirements from a single function of a single feature. The power of this framework isn't what it fixed — it's what it scales. Every feature in scope will generate its own set, and every gap it closes becomes a permanent improvement to the standard."
